# Deep Research: Multi-agent memory governance on 24GB Apple Silicon (July 2026)

**Date:** 2026-07-13 · **Method:** 104-agent deep-research workflow, 3-vote adversarial verification. Drives the memory-pressure-guardian v2 redesign (`scripts/memory-pressure-guardian.sh`).

## Summary

On a 24GB Apple Silicon Mac running macOS 26.x, the highest-leverage fix is Ollama configuration: set OLLAMA_MAX_LOADED_MODELS=1, OLLAMA_NUM_PARALLEL=1, a short OLLAMA_KEEP_ALIVE (e.g. 2m), and shrink num_ctx from 40960 — required KV-cache RAM scales as num_parallel * context, and any client can silently pin the model via a per-request keep_alive override, so the guard should also actively evict with `ollama stop` / keep_alive:0 and include a detector for the known wedged "Stopping..." unload (recovery = kill the ollama runner child process). For the trigger metric, poll `sysctl kern.memorystatus_vm_pressure_level` (1=normal, 2=warn, 4=critical) — this is the kernel's own signal, computed from available non-compressed memory with built-in rising/falling hysteresis — rather than free-RAM% or swap presence, which look alarming during healthy Apple Silicon use (though near-total swap exhaustion like 21.7/22.5GB remains a valid secondary critical trigger). macOS offers no supported per-process memory cap for enforcement: launchd's ResidentSetSize limit maps to RLIMIT_RSS which the kernel rejects (EINVAL, aliased to RLIMIT_AS), memorystatus_control is private/unstable, and Apple documents jetsam termination only for iOS-family platforms — so a user-space guard IS the enforcement layer, escalating from Ollama eviction to (speculatively) taskpolicy QoS demotion to SIGSTOP/kill. The action ladder must be Electron-aware: SIGSTOP does not fire Electron's render-process-gone/child-process-gone events (a stopped process hasn't "gone"), while SIGTERM/SIGKILL produces an observable 'killed' reason that lets the app respawn the helper under a new PID — explaining the observed Cursor-helper respawn, which is app/Chromium-level behavior, not documented Electron framework behavior — so prefer killing leaf workloads (Gradle daemons, ollama runner) over signaling Electron helper processes at all.

## Verified findings

### [high] Ollama keep-alive semantics give deterministic eviction, but any client can override them: OLLAMA_KEEP_ALIVE defaults to 5 minutes; keep_alive accepts duration strings, seconds, 0 (unload immediately), and negative values (pin indefinitely); `ollama stop <model>` unloads on demand. Critically, a per-request keep_alive on /api/generate or /api/chat OVERRIDES the server-level env var, so a co-located agent can silently pin the 8GB qwen3:8b model (e.g. keep_alive:-1) regardless of server config — open feature requests #10263/#11002 confirm there is no server-side cap. A guard should therefore verify residency via /api/ps and force-evict, not trust the env var.

*Evidence:* Ollama FAQ (fetched 2026-07-13): "By default models are kept in memory for 5 minutes before being unloaded... `keep_alive` accepts... Negative values to keep loaded indefinitely, 0 to unload immediately"; "The `keep_alive` API parameter with the `/api/generate` and `/api/chat` API endpoints will override the `OLLAMA_KEEP_ALIVE` setting." Both verified verbatim against the live primary source.
*Sources:* https://docs.ollama.com/faq, https://github.com/ollama/ollama/issues/10263, https://github.com/ollama/ollama/issues/11002

### [high] Required RAM scales multiplicatively as OLLAMA_NUM_PARALLEL * context length (KV-cache allocation), with OLLAMA_NUM_PARALLEL defaulting to 1 — so the incident's num_ctx=40960 with any parallelism >1 multiplies the KV-cache footprint. On a 24GB Mac, pin OLLAMA_NUM_PARALLEL=1 and reduce num_ctx (e.g. to 8192-16384) unless the workload genuinely needs 40k context.

*Evidence:* Ollama FAQ verbatim: "Required RAM will scale by OLLAMA_NUM_PARALLEL * OLLAMA_CONTEXT_LENGTH... a 2K context with 4 parallel requests will result in an 8K context and additional memory allocation." Mechanism corroborated by llama.cpp server docs (Ollama runs this engine): total KV cache must be per-slot context times parallel slots.
*Sources:* https://docs.ollama.com/faq, https://github.com/ggml-org/llama.cpp (upstream --ctx-size = T*P semantics)
*Votes:* 3-0

### [high] OLLAMA_MAX_LOADED_MODELS defaults to 3 per GPU (Apple Silicon Metal counts as 1 GPU, so default = 3), meaning an unconfigured Ollama on a 24GB Mac can hold up to three models resident "provided they fit in available memory." Setting OLLAMA_MAX_LOADED_MODELS=1 is the lever to prevent multi-model residency; note it caps concurrency but does not unload the single model (that is keep_alive's job).

*Evidence:* FAQ verbatim: "The maximum number of models that can be loaded concurrently provided they fit in available memory. The default is 3 * the number of GPUs or 3 for CPU inference." Checked against Ollama's Sept 2025 'New model scheduling' blog post — the default is unchanged.
*Sources:* https://docs.ollama.com/faq

### [high] The wedged 'Stopping...' unload is a real, known, recurring Ollama bug class on Apple Silicon macOS, and it is not definitively fixed. A guard should include a wedge detector: /api/ps showing Stopping... for >N minutes → pkill -f 'ollama runner'. Note: in that incident system resources were fine, so #14364 does not prove the wedge is memory-pressure-triggered.

### [high] macOS provides no supported per-process memory cap a guard can lean on. (a) memorystatus_control (since xnu 2107 / OS X 10.9) can set memorystatus/jetsam parameters but is private and undocumented — no public header, root/entitlement-gated, command semantics shift across versions (EINVAL reports on macOS 26.2). (b) launchd's HardResourceLimits/SoftResourceLimits are documented and map to setrlimit(2), but the ResidentSetSize key is dead text: on macOS 26.5.2 the SDK defines RLIMIT_RSS as a compatibility alias for RLIMIT_AS, and setrlimit(RLIMIT_RSS, ...) returns EINVAL for every finite value — empirically tested; the documented RSS cap does not constrain memory on current macOS.

*Evidence:* Levin: memorystatus_control is an "(undocumented) syscall" controlling memorystatus and jetsam. launchd.plist(5) verbatim: "Resource limits to be imposed on the job. These adjust variables set with setrlimit(2)" and "ResidentSetSize: The maximum size (in bytes) to which a process's resident set size may grow." Verifier empirically confirmed non-enforcement: RLIMIT_RSS is an alias for RLIMIT_AS and the kernel rejects finite values with EINVAL on macOS 26.5.2.
*Sources:* http://newosxbook.com/articles/MemoryPressure.html, https://www.manpagez.com/man/5/launchd.plist/, apple-oss-distributions/xnu (kern_memorystatus), local empirical test on macOS 26.5.2: setrlimit(RLIMIT_RSS) → EINVAL; sys/resource.h line 450
*Votes:* 3-0, 3-0, 3-0 (merged claims 8+10+11)

### [medium] Apple documents jetsam (memory-pressure termination with jetsam event reports) only for iOS, iPadOS, tvOS, visionOS, and watchOS — macOS is absent from the platform list, so jetsam-style killing of regular processes is not the documented macOS mechanism and a guard cannot rely on the kernel to reap memory hogs. Caveat: jetsam machinery exists in XNU on macOS and does fire in practice in extreme cases (agent SIGKILLs under swap exhaustion observed in this very fleet), but it is undocumented, unconfigurable by users, and fires far too late to protect interactivity — reinforcing that a user-space guard is the actual enforcement layer.

*Evidence:* Apple's jetsam-event-report doc lists exactly "iOS, iPadOS, tvOS, visionOS, and watchOS"; macOS appears nowhere on the page and the sample report is an iPad. Oakley corroborates jetsam as iOS-derived infrastructure appearing in macOS logs without documented regular-process killing. 2-1 vote reflects the documentation-vs-practice tension.

### [high] The right trigger metric for the guard is the kernel's own pressure level, not free RAM or swap presence. `sysctl kern.memorystatus_vm_pressure_level` returns 1 (NORMAL), 2 (WARN), or 4 (CRITICAL) on OS X 10.9+ (matching dispatch source.h bitmask 0x1/0x2/0x4; empirically reproduced on macOS 26.5.2 where the sysctl read 2/WARN while free RAM showed a healthy 31%). XNU computes this level from AVAILABLE_NON_COMPRESSED_MEMORY (active + inactive + free + speculative pages) crossing distinct rising/falling thresholds (Warning rises at 0.5 * AVAILABLE_MEMORY, falls at 1.2 * compact threshold) — i.e., the kernel already builds in hysteresis a 60s-interval guard can mirror, and raw free RAM (the incident's 165MB) is not the metric XNU itself uses. Prior art agrees: mac-memory-guard's README states free-RAM% and swap are "intentionally not triggers — on Apple Silicon both look alarming during perfectly healthy use." Recommended design: warn actions at level 2 sustained across 2 consecutive polls, critical ladder at level 4 or at near-total swap exhaustion (>90-95% of swap file used) as a secondary confirmatory trigger; require return to level 1 before releasing SIGSTOPs (hysteresis). Note the specific numeric thresholds are the author's synthesis, not sourced.

*Evidence:* Levin verbatim: "you can also query kern.memorystatus_vm_pressure_level, which is a 1 (NORMAL), 2 (WARN) or 4 (CRITICAL)". XNU doc verbatim: "AVAILABLE_NON_COMPRESSED_MEMORY = (active + inactive + free + speculative)" with the rising/falling threshold table; doc line 69 defines compact threshold = 0.5 * AVAILABLE_MEMORY on macOS. mac-memory-guard README quoted verbatim. Verifier live-reproduced level 2 alongside 31% free RAM, demonstrating divergence from free-RAM%. Important: a companion claim that the sysctl exposes five levels (0-4) was REFUTED 0-3 — treat the values strictly as 1/2/4.
*Sources:* http://newosxbook.com/articles/MemoryPressure.html, https://github.com/apple-oss-distributions/xnu/blob/main/doc/vm/memorystatus_notify.md, https://github.com/TomGranot/mac-memory-guard, empirical verification on macOS 26.5.2 (sysctl + memory_pressure)

### [high] Electron-safe action ladder rules: (a) SIGSTOP is invisible to Electron's app-level handlers — 'render-process-gone' / 'child-process-gone' fire only when a process "unexpectedly disappears... crashed or killed"; a stopped process has not gone, so STOP alone triggers no handler (it manifests only as unresponsiveness). (b) SIGTERM/SIGKILL of a helper IS observable: the details.reason enum distinguishes 'killed' ("Process was sent a SIGTERM or otherwise killed externally") from 'crashed'/'oom', giving the parent app a documented hook to relaunch the helper under a new PID. (c) Electron itself documents NO automatic framework-level respawn — the observed Cursor-helper respawn under new PIDs after being STOPped/killed is app-level (Cursor's own handlers) or Chromium-internal (GPU/utility process relaunch), not documented Electron behavior. Guard implication: never SIGSTOP or kill Electron helper processes directly (STOP risks IPC wedges and watchdog side effects; kill just triggers respawn under a new PID, defeating the guard); instead target the leaf workloads they spawn — Gradle daemons, ollama runner, node build processes — and allowlist *Helper*, *Renderer*, WindowServer, and anything with an Electron/Chromium parent.

*Evidence:* Electron app API verbatim: "Emitted when the renderer process unexpectedly disappears. This is normally because it was crashed or killed"; reason enum includes 'killed' — "Process was sent a SIGTERM or otherwise killed externally" — distinct from 'crashed' and 'oom'; no state in the enum corresponds to suspension. electron#43020 shows a suspended process presents as unresponsiveness, not process-gone. electron#19887/#23291 show respawn must be implemented by the host app. Note: a separate claim that Chromium's hang detector documents SIGSTOP-tripping-watchdogs was REFUTED 0-3, so the IPC-wedge risk of STOPping Electron helpers remains plausible-but-undocumented.
*Sources:* https://www.electronjs.org/docs/latest/api/app, https://www.electronjs.org/docs/latest/api/structures/render-process-gone-details, https://github.com/electron/electron/issues/43020, https://github.com/electron/electron/issues/19887

## Caveats

Several requested research areas produced NO verified claims and remain unanswered: taskpolicy(8) -c/-b semantics (child-process propagation to Gradle daemons, E-core scheduling effects), Apple's June-2025 containerization framework as a memory-cap mechanism for CLI agents, purpose-built 2025-2026 multi-agent governance tooling, and Cursor-specific controls for background agents spawning local builds — treat any statement about these as speculative. Seven claims were refuted in verification and must not be relied on, notably: the five-level (0-4) pressure enum (use 1/2/4 only), the assertion that mac-memory-guard triggers exclusively on the pressure sysctl, the Chromium hang-detector/SIGSTOP documentation (so the 'SIGSTOP wedges Electron IPC' pitfall is plausible folklore, not documented), and the claim that macOS jetsam only kills idle-exit-marked processes. The Ollama wedge bug (#14364) is version-scoped to 0.13.3 and was NOT shown to be memory-pressure-triggered — do not frame it as a pressure-induced failure; a collaborator says newer releases fixed prior instances, so upgrade first. The runner-kill workaround is maintainer-recommended ('should restore function') but was not empirically confirmed in the thread. Jetsam-on-macOS is a documentation-vs-practice split (2-1 vote): it demonstrably fires in extreme swap exhaustion but is undocumented and unconfigurable. Time-sensitivity: Ollama defaults (MAX_LOADED_MODELS=3, NUM_PARALLEL=1) reflect docs fetched 2026-07-13 and have changed historically (0.2-0.5 auto-selected parallelism); re-verify after Ollama upgrades. All numeric thresholds in the guard redesign (2 polls, swap >90%, num_ctx <=16384, KEEP_ALIVE=2m) are synthesis, not sourced.

## Refuted claims (do not rely on)

- On macOS (OS X, as of 10.9 Mavericks), the memorystatus/jetsam mechanism does NOT kill arbitrary processes under memory pressure as on iOS; it kills only processes explicitly marked for idle exit. This means jetsam-style involuntary killing of ordinary user processes (e.g., agent CLIs, Gradle daemons) is not the macOS desktop behavior — a guard cannot rely on the kernel to reap memory hogs.
- The only other memory-related rlimits launchd exposes are the data segment (sbrk break) and stack segment sizes, which do not bound total footprint of modern mmap/malloc-based processes like Gradle JVMs or Electron helpers — so launchd.plist offers no reliable whole-footprint memory cap for agent processes.
- On the jetsam-supported platforms, the per-process resident memory limit that makes a process eligible for termination is imposed by the system on all apps — Apple describes it as a fixed system-imposed limit, not something a user or developer configures per process.
- mac-memory-guard triggers exclusively on the kernel memory-pressure level read via sysctl kern.memorystatus_vm_pressure_level (matching Activity Monitor's green/yellow/red), and deliberately treats swap usage and free-RAM percentage as display-only, never as triggers — directly relevant to which metric combination a 24GB Apple Silicon guard should key on.
- Its process attribution sums per-process RSS aggregated per .app bundle and is restricted to user-facing apps under /Applications and /Users — an allowlist approach that would inherently miss CLI agents (Claude/Codex CLI, Gradle daemons, ollama llama-server), mirroring the 'agents=0' detection failure in the incident.
- Chromium's own hang detector is disabled on development systems specifically because a process stopped by a debugger (i.e., in a SIGSTOP-like stopped state) is indistinguishable from a hung process — documenting that externally-stopped Chromium processes trip liveness/hang watchdogs, a known pitfall for SIGSTOP-based guards pausing Electron/Chromium helpers.
- kern.memorystatus_vm_pressure_level exposes five states, not just normal/warn/critical: Normal (0), Warning (1), Urgent (2, synonymous with Warning), Critical (3), and Jetsam (4, visible to kernel threads only) — so a guard polling this sysctl should treat 1/2 as warn and 3 as critical.
